\tightsection{Part 2: Technical Defensive Response}
\textbf{Selected risk.} R1, unauthorised access to assumed production backup data on \texttt{backup01} via suspected compromise of the \texttt{backup} service account, reached through opportunistic SSH password guessing from \texttt{203.0.113.89} and extended toward host-level control through the account's sudo rights.

\textbf{Defensive goal.} The defence targets the full path rather than brute force alone:
\[
\text{SSH brute-force attempts} \rightarrow \text{successful backup login} \rightarrow \text{active session} \rightarrow \text{sudo as root}.
\]
The response is layered: prevention to stop the guessing from succeeding, containment to limit damage if a credential is already compromised, and detection to catch any success the preventive layer misses.

\textbf{Cyber Kill Chain framing.} We use the model as a defensive-interruption table for the observed SSH path, not as a claim that every Cyber Kill Chain stage occurred.

The log shows that multiple remote source labels reached SSH, but it does not reveal the network path or existing firewall policy. Figure~\ref{fig:backup-firewall-design} therefore presents a proposed design rather than a claim about the original environment. Under the stated production-backup assumption, \texttt{backup01} only needs to communicate with systems that provide backup data and with managed hosts used for administration. Both should be on the internal LAN or enter through an approved internal access path; accepting inbound SSH from untrusted networks provides no necessary backup function. It instead exposes a host containing production copies and recovery data to automated scanning and credential attacks. A default-deny firewall would remove that unnecessary path and permit only approved internal systems to reach the server. This protects not only the host but also the organisation's ability to restore production data after an incident. If off-site replication is required, it should use a separately approved outbound connection rather than direct inbound SSH.

Network filtering should be backed by key-only authentication. Disabling \texttt{PasswordAuthentication} removes the observed guessing-to-success path, while rate limiting, for example with fail2ban, reduces the load from repeated attempts that reach the internal service. \texttt{PermitRootLogin no} should also be enforced as a precaution, although the failed root logins do not show whether direct root access was enabled.

Key-based authentication does not remove the risk of a stolen key. Since \texttt{backup} is assumed to be a dedicated service account, its key should invoke a forced backup command rather than open a normal shell. Its sudo permissions should contain only the commands required by the backup process and should exclude unrestricted sudo, \texttt{su}, shell escapes, package management, and general system administration. Administrators can use the named accounts already present in the log, with a separate audited break-glass account for exceptional access. These restrictions preserve the backup process without retaining the general sudo path visible at 03:08:19.

SELinux can provide a second containment boundary by placing the backup process in a confined domain. That domain should permit access only to the data, devices, and network resources required for backup operations, while denying writes to account databases, sudoers, cron locations, SSH host keys, and unrelated logs. This protection depends on the approved command path remaining confined; an unrestricted transition to an unconfined root domain would bypass it. SELinux therefore complements, rather than replaces, restrictive sudo policy.

Detection should use both approved-source information and recent authentication behaviour. Once the organisation records which systems are allowed to run backups, an accepted \texttt{backup} login from any other source should trigger an alert. This allowlist is a proposed control and is not inferred from the current log. A second rule should correlate repeated failures with a later successful login. A simple ``failed then accepted'' condition is too broad because thirteen sources match it, whereas the preceding-hour account baseline clearly separates \texttt{203.0.113.89} from established sources.

Accordingly, a high-priority alert should be raised if an accepted \texttt{backup} login originates outside the approved source set, or if one source records at least 50 failed \texttt{backup} passwords within 60 minutes before an accepted login. Severity should increase when the source has not authenticated previously, the event occurs during the 00:00--05:59 attack window, or same-user sudo commands appear at the same timestamp.

Applied to this extract, the behavioural rule flags only \texttt{203.0.113.89}: 87 preceding-hour failures against an established-source maximum of three, followed by the source's first accepted \texttt{backup} login. The result combines volume, target sensitivity, source novelty, timing, and privileged follow-on activity without treating any one field as proof of compromise.

Together, these controls address different parts of the observed path. The firewall blocks untrusted access before authentication, key-only SSH removes the password-guessing route, and source and command restrictions limit the use of a stolen key. Restrictive sudo policy and SELinux reduce the effect of a compromised \texttt{backup} account, while correlation alerts and central logging improve detection and investigation.

The design still has practical limitations:
\begin{itemize}
    \item Firewall and source restrictions require an accurate inventory and change process. A compromised approved system would pass the source check, while a stale rule could interrupt backups or administration.
    \item Detection selectivity depends on tuning. The 50-in-60-minute threshold fits this extract but may miss low-and-slow or distributed attempts and must be recalibrated against production history.
    \item Rate-limiting/lockout can cause denial of service. Aggressive fail2ban thresholds could block legitimate operations or be abused to lock out accounts; thresholds must be tested so scheduled backup jobs are not interrupted.
    \item Key management overhead. Key-only SSH shifts risk to key storage and rotation; a stolen key still authenticates unless source- and command-restricted.
    \item SELinux is not absolute. Policy authoring and maintenance are non-trivial, mislabelling can break services, and an attacker who reaches an unconfined admin role could still relabel or disable enforcement. It raises the cost of compromise rather than eliminating it.
\end{itemize}

These controls cannot establish what happened after the Jul 11 login. Confirming or clearing the event still requires the wider evidence review recommended in Part 1, including shell history, the full journal, audit logs, cron records, and file-integrity checks.
